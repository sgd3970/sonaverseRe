# Sonaverse 리뉴얼 – 코드 작성 착수 체크리스트

> **목적**: 실제 리뉴얼 코드를 쓰기 전에, **반드시 확인해야 할 최소 항목**을 한 장으로 정리  
> 이 체크리스트만 통과하면 바로 구현을 시작해도 좋다는 기준선

---

## 1. 설계·문서 관련

- [ ] 구현하려는 페이지/기능이 **어느 단계(1~7단계)**인지 `IMPLEMENTATION_PLAN.md`에서 확인했다.
- [ ] 관련되는 데이터/도메인이 `DATABASE_SCHEMA`·`API_ROUTES` 문서 어딘지 파악했다.  
      (※ 현재 문서는 삭제되었으면, 새로운 스키마/라우트 정의 시 다시 생성 예정)
- [ ] 필요한 UI 패턴이 `COMPONENT_MAPPING.md`에 정의되어 있는지 확인했다.  
      - 이미 있다 → 재사용 또는 약간의 확장  
      - 없다 → 이름 규칙에 맞춰 새 컴포넌트 후보를 잡았다.
- [ ] 이 기능이 어느 폴더(`app`, `features`, `shared`, `lib`)에 위치할지 `FOLDER_STRUCTURE_GUIDE.md`로 정리했다.

---

## 2. 성능·이미지·데이터 관련

- [ ] 이 기능에서 LCP에 영향을 줄 수 있는 요소(히어로 이미지, 큰 컴포넌트)를 파악했다.
- [ ] 필요한 경우:
  - [ ] 이미지 로딩 전략(`next/image`, `priority`, `lazy`)을 결정했다.
  - [ ] Blur 플레이스홀더(`plaiceholder`) 적용 대상인지 판단했다.
- [ ] API 호출/데이터 페칭이 있다면:
  - [ ] 서버 컴포넌트/클라이언트 컴포넌트 역할을 구분했다.
  - [ ] 불필요한 필드를 가져오지 않도록 설계했다.

---

## 3. 폼·검증·보안 관련

해당 기능에 폼 또는 입력이 있다면:

- [ ] `zod` 스키마가 정의되어 있다 (`tools/TOOL_ZOD.md` 참고).
- [ ] 폼은 `react-hook-form` 기반으로 설계했다 (`tools/TOOL_REACT_HOOK_FORM.md` 참고).
- [ ] 스팸/봇 공격 가능성이 있는 엔드포인트(문의, 로그인 등)는 `rate-limiter-flexible` 적용 여부를 검토했다.

---

## 4. 도구·품질 관련

- [ ] 이 기능이 머지되면, **어떤 도구로 검증할지** 미리 생각했다.
  - [ ] 번들 크기 영향이 크면 → `@next/bundle-analyzer` (`TOOL_BUNDLE_ANALYZER.md`)
  - [ ] 성능/접근성 영향이 크면 → Lighthouse / LHCI (`TOOL_LIGHTHOUSE_CI.md`)
  - [ ] 안정성이 중요하면 → 최소 1개 이상의 테스트 (`TOOL_VITEST_TESTING_LIBRARY.md`)
  - [ ] 장애 발생 시 영향이 크면 → Sentry에서 확인할 지점 (`TOOL_SENTRY_NEXTJS.md`)

---

## 5. PR 준비 관련

- [ ] 이 작업을 **하나의 브랜치/PR로 묶었을 때 범위가 너무 크지 않은지** 확인했다. (`WORKFLOW_RULES.md`)
- [ ] PR 설명 초안에 다음 내용을 적을 수 있다:
  - 이 PR의 목적 (한 문장)
  - 변경되는 페이지/기능 이름
  - 예상되는 성능/UX 영향
  - 관련 문서 링크 (있다면)

---

## 6. 요약

이 체크리스트는 “실제 코드 작성 전에 5분만 투자”하는 용도로 사용한다:

1. **문서 위치 확인** – 내가 작업할 기능이 어디에 해당하는지 파악  
2. **성능/UX 영향 상상** – LCP, 이미지, 데이터, 폼/보안  
3. **검증 도구 상상** – 나중에 무엇으로 품질을 확인할지 미리 정해두기  

이 세 가지만 매번 반복하면, 리뉴얼 프로젝트 전반의 품질과 일관성이 크게 좋아진다.
