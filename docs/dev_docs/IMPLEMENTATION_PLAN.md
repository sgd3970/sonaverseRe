# Sonaverse 리뉴얼 구현 계획 – 페이지 우선순위 및 단계별 진행

> **목적**: 리뉴얼 Next.js 프로젝트 구현 시, **어떤 페이지를 어떤 순서로 구현할지**를 고정해 개발 흐름을 일정하게 유지하기 위함  
> **전제**: 속도(성능) 최우선, 관리자 전용 로그인, 기존 `sonaverse`의 기능·콘텐츠를 기반으로 함

---

## 1. 구현 단계 개요

1. **1단계 – 공통 레이아웃 & 디자인 시스템 반영**
2. **2단계 – 공개 메인 플로우 페이지**
3. **3단계 – 콘텐츠 리스트/상세 페이지**
4. **4단계 – 문의 플로우**
5. **5단계 – 관리자 로그인 & 대시보드**
6. **6단계 – 관리자 콘텐츠 관리 페이지**
7. **7단계 – 성능·테스트·모니터링 보강**

---

## 2. 1단계 – 공통 레이아웃 & 디자인 시스템

### 2.1 대상

- 공통 레이아웃:
  - `RootLayout` (`app/layout.tsx`)
  - 공통 헤더
  - 공통 푸터
- 공통 UI 토큰:
  - 컬러, 타이포, spacing, radius 등 (Stitch Design System 기준)

### 2.2 목표

- PC·모바일 공통 **헤더/푸터** 구조 고정
- Design System에 정의된:
  - Primary/Secondary 버튼
  - 카드 레이아웃
  - 섹션 헤더
  - 그리드/레이아웃 패턴
  를 컴포넌트로 1차 구현

---

## 3. 2단계 – 공개 메인 플로우 페이지

### 3.1 우선순위 페이지

1. `/` – **홈 페이지**
2. `/products/manbo-walker` – **만보 제품 상세**
3. `/products/bodeum-diaper` – **보듬 제품 상세**

### 3.2 각 페이지 핵심 요소

- **`/` 홈**
  - 히어로 섹션 (LCP 대상 이미지, 핵심 카피)
  - 문제 정의 / 솔루션 소개 섹션
  - 제품 진입 섹션 (만보/보듬 카드)
  - Sonaverse Story, Press 섹션 진입용 카드

- **`/products/manbo-walker`**
  - 제품 히어로(이미지 + 제품 USP)
  - 기능 섹션 (Stitch 디자인에 맞춘 아이콘/카드)
  - 사용 시나리오/타겟 사용자 설명
  - 관련 스토리/언론보도 CTA

- **`/products/bodeum-diaper`**
  - 제품 라인업 리스트 (카테고리/옵션)
  - 각 제품 군(테이프형/팬티형 등) 카드
  - 관련 스토리/언론보도 CTA

---

## 4. 3단계 – 콘텐츠 리스트/상세 페이지

### 4.1 우선순위 페이지

4. `/sonaverse-story` – **소나버스 스토리 리스트**
5. `/press` – **언론보도 리스트**

### 4.2 각 페이지 핵심 요소

- **`/sonaverse-story`**
  - 카테고리 필터 (데스크톱: 탭/피벗, 모바일: 슬라이드)
  - 메인 스토리(대표 카드)
  - 일반 스토리 카드 그리드
  - 페이지네이션 or “더 보기” 로딩

- **`/press`**
  - 연도/카테고리 필터 (필요 시)
  - 언론보도 카드 리스트 (썸네일, 언론사, 날짜)
  - 외부 링크 / 상세 페이지 연결

> 상세 페이지(`/sonaverse-story/[slug]`, `/press/[slug]`)는 3단계 말~4단계 초에 함께 구현할 수 있다.

---

## 5. 4단계 – 문의 플로우

### 5.1 대상 페이지

6. `/inquiry` – **구매·제휴 문의 페이지**

### 5.2 핵심 요소

- 문의 유형 선택 (서비스 소개 / 제휴 / 구매 / 기타)
- 기본 정보 입력 (이름, 직함, 회사명, 연락처, 이메일)
- 상세 문의 내용
- 파일 첨부 (제한된 포맷/크기)
- 개인정보 수집·이용 동의 체크박스
- 제출 후 완료 화면/토스트

> 이 단계에서 `zod` + `react-hook-form` + `rate-limiter-flexible` 적용을 1차로 묶어 구현한다.

---

## 6. 5단계 – 관리자 로그인 & 대시보드

### 6.1 대상 페이지

7. `/admin/login` – **관리자 로그인 페이지**  
8. `/admin` – **관리자 대시보드 (요약)**  

### 6.2 핵심 요소

- `/admin/login`
  - 이메일/비밀번호 입력
  - 로그인 실패/잠금 처리
  - Rate Limit 적용

- `/admin`
  - 최근 언론보도/스토리/문의 요약 카드
  - 빠른 작업 진입 버튼 (새 언론보도 등록, 새 스토리 작성 등)

---

## 7. 6단계 – 관리자 콘텐츠 관리 페이지

### 7.1 대상 페이지 (예시)

- `/admin/press` – 언론보도 목록/검색
- `/admin/press/new` – 언론보도 등록
- `/admin/press/[id]` – 언론보도 수정
- `/admin/stories` – 스토리 목록/검색
- `/admin/products` – 제품 관리
- `/admin/inquiries` – 문의 목록/상세/상태 변경

> 실제 URL/구조는 `API_ROUTES` 및 `DATABASE_SCHEMA`에 맞춰 조정.

---

## 8. 7단계 – 성능·테스트·모니터링 보강

### 8.1 도구/작업

- 번들 분석 (`@next/bundle-analyzer`)
- Lighthouse CI (`@lhci/cli`)
- Vercel Analytics & Speed Insights
- Sentry 연동
- Vitest + Testing Library로 핵심 경로 테스트 추가

### 8.2 적용 우선순위

1. 성능 측정 도구 (Lighthouse CI, Vercel Speed Insights)
2. 에러/로그 수집 (Sentry)
3. 테스트 (문의/로그인/리스트 페이지 최소 1케이스씩)

---

## 9. 요약 타임라인 (추천)

- **Week 1–2**: 공통 레이아웃 + 홈/제품 페이지
- **Week 3**: 스토리/언론보도 리스트 + 일부 상세
- **Week 4**: 문의 페이지 + 관리자 로그인/대시보드
- **Week 5**: 관리자 콘텐츠 관리 페이지 1차
- **Week 6**: 마무리 성능 튜닝·테스트·모니터링

> 실제 일정은 팀 리소스와 우선순위에 따라 조정하되, **공개 사용자 플로우 → 관리자 플로우 → 품질 보강** 순서를 유지하는 것을 추천한다.
